# アドバイザー戦略（Advisor Strategy）

参考: https://claude.com/blog/the-advisor-strategy

## 概念

| 役割 | モデル | 担当 |
|------|--------|------|
| **Executor（実行者）** | Sonnet（現在のモデル） | タスク全体を実行・ファイル操作・ビルド・テスト |
| **Advisor（助言者）** | Opus | 判断が難しい局面でのみ相談を受ける |

Opus はツール呼び出しや最終出力を行わない。計画・方針・修正案を提供するだけ。

## Opus に相談するタイミング

以下のいずれかに該当するとき、Agent ツール（`model: "opus"`）でアドバイザーを呼び出す：

| 状況 | 例 |
|------|-----|
| アーキテクチャ判断 | CommonLibrary に入れるべきか、呼び出し側に書くべきか |
| 複数の実装方針のトレードオフ比較 | リポジトリパターン vs 直接 DBContext |
| セキュリティ上の判断 | 認可ロジックの設計、CSRF 対策の範囲 |
| 根本原因が特定できない複雑なバグ | エラーの原因仮説を並列で検討したい |
| スプリントコントラクトの完成条件の妥当性確認 | 抜け漏れがないか第三者視点で確認 |

## Opus に相談しないタイミング

- ルーティンな実装（CRUD・ビュー生成・DI 登録）
- ビルドエラーの修正（スタックトレースが明確な場合）
- ファイルの読み書き・検索
- コミット・ブランチ操作

## 呼び出し方

```
Agent ツール:
  subagent_type: (省略 → general-purpose)
  model: "opus"
  prompt: |
    以下の判断について助言をください。
    実装はしなくてよい。方針・理由・代替案を返してください。

    状況: [現在の文脈]
    検討している選択肢:
    1. ...
    2. ...
    判断したいこと: [具体的な問い]
```

## 効果（ベンチマーク実績）

- Sonnet + Opus アドバイザー は Sonnet 単独より約 2.7pt 向上
- タスクあたりコストは 11.9% 削減
- Haiku + Opus アドバイザー は BrowseComp で 19.7% → 41.2% に改善

## 原則

> **「Sonnet が走り、Opus が考える」**
>
> Executor がすべての実行を担当し、壁に当たったときだけ Advisor に判断を仰ぐ。
> Advisor の助言を受けたら、実行は必ず Executor（Sonnet）が行う。
