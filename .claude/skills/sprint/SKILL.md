実装を開始する前に、スプリントコントラクト（完成条件）を宣言してユーザーの承認を得ます。

## コントラクトの作成

ユーザーの依頼内容を整理し、以下のフォーマットで提示する：

```
【スプリントコントラクト】
実装内容：〇〇機能を追加する

完成条件：
- [ ] （正常系）〇〇したとき〇〇になる
- [ ] （権限・認可）〇〇ロールのみ〇〇できる / 未ログインはリダイレクトされる
- [ ] （異常系）〇〇のとき〇〇エラーが表示される / バリデーションメッセージが出る
- [ ] （副作用なし）既存の〇〇機能に影響がない

→ 上記の完成条件に問題がなければ実装を開始します。
```

## 条件の具体化ガイド

| 観点 | 良い例（具体的） | 悪い例（曖昧） |
|------|--------------|-------------|
| 正常系 | 「登録ボタンを押すと一覧に追加される」 | 「登録できる」 |
| 認可 | 「Admin のみ削除ボタンが表示される」 | 「権限制御がある」 |
| 異常系 | 「必須項目未入力でバリデーションエラーが赤字表示される」 | 「エラー処理がある」 |
| 副作用 | 「UserManagement 画面のテストが引き続き通る」 | 「既存機能に影響しない」 |

## 承認後のルール

- 承認されたら実装を開始する
- **完成条件は実装途中で変えない**。変えたい場合はユーザーに確認してから変更する
- agent-browser でのブラウザ確認時に、この完成条件リストをそのまま検証チェックリストとして使う

## 適用タイミング

| 状況 | 使う？ |
|------|-------|
| 新機能・新ページを実装するとき | 必ず使う |
| バグ修正で影響範囲が広いとき | 使う |
| リファクタリングで複数ファイルを変更するとき | 使う |
| 1ファイルの軽微な修正 | スキップ可 |
