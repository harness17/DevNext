# DevNext 設計思想ガイド

## はじめに

DevNext は、新規案件の出発点となる ASP.NET Core テンプレートです。このドキュメントは、DevNext に組み込まれた設計判断の背景・代替案・採用理由をまとめたものです。面接やポートフォリオ説明で「なぜこう設計したか」を説明するときに参考にしてください。

**対象読者**: Web 開発者・SIer 案件従事者・就活生  
**想定質問**: 「このプロジェクトの設計で工夫した点は？」「なぜこの技術スタックを選んだ？」

## 1. 認証・ユーザー管理

### 背景

どのWeb アプリケーションにも認証は必須。ユーザー管理を手実装すると以下のリスクが生じます：
- パスワードハッシュの実装ミス → セキュリティ侵害
- セッション・トークン管理の不備 → セッションハイジャック
- ブルートフォース対策を忘れる → パスワード総当たり攻撃を許す
- ロールベース認可の実装漏れ → 水平権限昇格（IDOR）

### 代替案

| 方式 | 長所 | 短所 |
|------|------|------|
| **ASP.NET Core Identity（採用）** | フレームワークが提供、セキュリティベストプラクティス内蔵、テスト済み | 学習曲線がある、カスタマイズに時間がかかる場合がある |
| Auth0 / Okta（外部SaaS） | 保守が外部、スケール対応が容易 | 外部依存、追加コスト、ネットワーク遅延 |
| 手実装 JWT | シンプル、カスタマイズ自由度が高い | セキュリティ実装の責任が全部自分、テスト不足になりやすい |
| Session（サーバーサイドSession） | 古くから実績あり | マルチサーバー展開時に状態管理が複雑 |

### 採用理由

- **フレームワームネイティブ**: ASP.NET Core の公式提供なので、MVC とのシームレスな統合が可能
- **セキュリティデフォルト**: パスワードポリシー・ロックアウト・トークン更新が初期実装済み
- **テスト済みコード**: セキュリティベストプラクティスが組み込まれているので手実装のリスクを回避
- **新案件への転用性**: テンプレートとして使うので、次の案件でもそのまま活用できる

### 実装の工夫

```csharp
// Program.cs での設定例
builder.Services.AddIdentity<ApplicationUser, ApplicationRole>(options =>
{
    options.Password.RequiredLength = 6;
    options.Password.RequireDigit = true;
    options.Password.RequireUppercase = true;
    options.Password.RequireLowercase = true;
    options.Password.RequireNonAlphanumeric = true;
    
    options.Lockout.DefaultLockoutTimeSpan = TimeSpan.FromMinutes(5);
    options.Lockout.MaxFailedAccessAttempts = 5;
})
.AddEntityFrameworkStores<SiteDbContext>()
.AddDefaultTokenProviders();
```

**ポイント**:
- パスワード最低6文字 + 大小文字・数字・記号すべて必須
- ブルートフォース対策: 5回失敗で5分ロックアウト

## 2. データモデル設計: SiteEntityBase 継承

### 背景

複数のテーブルを管理していると、以下の問題が起こります：
- 各テーブルで `CreatedAt`, `UpdatedAt`, `CreatedBy` をバラバラに定義
- 監査ログが必要なテーブルと不要なテーブルで構造がばらばら
- テーブルごとに ID の型（`int` or `long`）が異なる
- 「いつ誰が何を変更したか」を追跡できない

### 代替案

| 方式 | 長所 | 短所 |
|------|------|------|
| **SiteEntityBase 基底クラス（採用）** | 共通カラムを統一、監査ログが一貫性を持つ | 階層構造が深くなる場合がある |
| マッピング・テーブル（Audit Table） | 履歴を専用テーブルで管理、本体テーブルはシンプル | 履歴取得時に JOIN 複数回が必要 |
| Shadow Properties（EF Core） | コード上は見えないが DB には作成 | マイグレーションで可視性が低い |
| 各テーブルで手動定義 | シンプル | 不整合が起こりやすい、保守負荷大 |

### 採用理由

- **統一性**: すべてのエンティティが同じ監査カラムを持つので、リポジトリの WHERE 句を統一できる
- **拡張性**: 新しいテーブルを追加するときに基底クラスを継承するだけで完成
- **監査対応**: 「いつ誰が何を作成・更新したか」が常に追跡可能
- **削除方式の統一**: 論理削除が必要なテーブルに `IsDeleted`, `DeletedAt` を付与できる

### 実装の工夫

```csharp
// 基底クラス
public abstract class SiteEntityBase : EntityBase
{
    public long CreatedById { get; set; }
    public long? UpdatedById { get; set; }
}

// 監査カラムが必要なエンティティ
public class ApprovalRequestEntity : SiteEntityBase
{
    public string Title { get; set; }
    // 履歴が必要な場合
}

// 履歴パターン（更新の過程を記録する場合）
public abstract class ApprovalRequestEntityBase : SiteEntityBase
{
}

public class ApprovalRequestEntity : ApprovalRequestEntityBase
{
    // 本体
}

public class ApprovalRequestEntityHistory : ApprovalRequestEntityBase, IEntityHistory
{
    public long HistoryId { get; set; }
    // 履歴フラグ
}
```

**ポイント**: ID は統一して `long` 型にする（Integer overflow を回避）

## 3. セキュリティ対策

### 3.1 IDOR（水平権限昇格）対策

#### 背景

「ログイン済み」≠「このリソースにアクセスできる」という原則を忘れると IDOR 脆弱性が生まれます。

```csharp
// NG: ログインチェックのみで所有権チェックなし
[Authorize]
public IActionResult EditRequest(long id)
{
    var req = _repo.GetById(id); // 他人のデータも取得可能
    return View(req);
}
```

#### 対策コード

```csharp
// OK: 所有権チェックを追加
[Authorize]
public IActionResult EditRequest(long id)
{
    var userId = _userManager.GetUserId(User);
    var req = _repo.GetById(id);
    
    // ユーザーが自分のリソースを操作しているか確認
    if (req == null || req.CreatedById != userId)
        return Forbid(); // 403 Forbidden
    
    return View(req);
}
```

#### 代替案

| 方式 | 長所 | 短所 |
|------|------|------|
| **コントローラー内チェック（採用）** | 各エンドポイントで明示的に制御、テストしやすい | チェック漏れのリスク |
| カスタム Authorization Attribute | 属性化でコードがスッキリ | 複雑な認可ロジックには向かない |
| リポジトリレイヤーで自動フィルタリング | 呼び出し側が何もしなくても安全 | 誤ってフィルタをバイパスしやすい |

### 3.2 CSRF（クロスサイトリクエストフォージェリ）対策

#### 背景

`<form>` 送信や Ajax は同一オリジン・ヘッダーで自動送信されるため、攻撃者サイトからの POST リクエストを防ぐ必要があります。

#### 対策コード

```html
<!-- Razor View で自動的に CSRF トークンを埋め込む -->
<form method="post">
    @Html.AntiForgeryToken()
    <!-- 他のフォーム要素 -->
    <button type="submit">送信</button>
</form>
```

```csharp
// コントローラー側
[Authorize]
[ValidateAntiForgeryToken] // 属性を付けるだけで検証自動
public IActionResult CreateRequest(RequestViewModel model)
{
    // ...
}
```

### 3.3 XSS（クロスサイトスクリプティング）対策

#### 背景

ユーザー入力を HTML に出力するときに、スクリプトタグをそのまま埋め込むと XSS 攻撃を許します。

#### 対策コード

```razor
<!-- NG: HTML エンコードなし -->
@Html.Raw(userInput)

<!-- OK: 自動エンコード -->
@userInput

<!-- JSON 値として安全に出力 -->
var message = @Html.Raw(JsonSerializer.Serialize(userInput));
// JavaScript 側で .text() で安全に処理
element.text(message);
```

## 4. リポジトリパターン

### 背景

DbContext を直接コントローラー・サービスに注入すると：
- テスト時にモックしにくい（DbContext はモック困難）
- データアクセスロジックが散在
- DB 操作の統一性が失われる

### 代替案

| 方式 | 長所 | 短所 |
|------|------|------|
| **リポジトリパターン（採用）** | データアクセスを一元化、テストしやすい | リポジトリ層の保守負荷 |
| DbContext 直接利用 | シンプル | テストが難しい、テスト時に実 DB が必要 |
| Entity Framework Specifications | ビルダーパターンで複雑クエリを読みやすく | 学習曲線がある |

### 採用理由

- **テスト可能性**: モック Repository を DI できるので、DB を使わないユニットテストが書ける
- **責務分離**: SQL ロジック をリポジトリに集約、サービスはビジネスロジックに専念
- **再利用性**: CRUD 基本操作を `RepositoryBase` に実装

### 実装例

```csharp
// リポジトリ基底
public abstract class RepositoryBase<TEntity> : IRepository<TEntity>
    where TEntity : class, IEntity
{
    protected readonly DbContext _context;
    
    public async Task<TEntity> GetByIdAsync(long id)
    {
        return await _context.Set<TEntity>()
            .FirstOrDefaultAsync(e => e.Id == id);
    }
    
    public async Task AddAsync(TEntity entity)
    {
        _context.Set<TEntity>().Add(entity);
        await _context.SaveChangesAsync();
    }
}

// 具体的なリポジトリ
public class ApprovalRequestRepository : RepositoryBase<ApprovalRequestEntity>
{
    public async Task<List<ApprovalRequestEntity>> GetByUserAsync(long userId)
    {
        return await _context.ApprovalRequests
            .Where(r => r.CreatedById == userId)
            .OrderByDescending(r => r.Id)
            .ToListAsync();
    }
}
```

## 5. DI（依存性注入）パターン

### 背景

サービスやリポジトリを `new` で直接生成すると：
- テスト時に実装を差し替えられない
- 依存関係が複雑に絡み合う
- 新しい依存を追加するときにコンストラクタ引数が増える

### 採用理由

```csharp
// NG: new で直接生成
public class NotificationService
{
    private readonly NotificationRepository _repo = new();
}

// OK: コンストラクタで注入
public class NotificationService
{
    private readonly NotificationRepository _repo;
    
    public NotificationService(NotificationRepository repo)
    {
        _repo = repo;
    }
}
```

### Program.cs での登録

```csharp
builder.Services.AddScoped<Site.Repository.ApprovalRequestRepository>();
builder.Services.AddScoped<Site.Service.ApprovalRequestService>();
builder.Services.AddScoped<AccessLogAttribute>();
```

**ポイント**: AddScoped（リクエストごとに生成）を原則に、メモリリーク・スレッドセーフティを確保

## 6. CommonLibrary 責務分離

### 背景

新案件で共有できるコードとプロジェクト固有のコードを混ぜると：
- 他の案件で再利用しようとしたときに依存を剥がせない
- テンプレート更新時に競合が多発
- CommonLibrary がどんどん肥大化

### 採用理由

```
DevNext/
├── CommonLibrary/       ← 再利用可能なもの
│   ├── Entity/
│   │   └── SiteEntityBase, EntityBase, IEntity
│   ├── Repository/
│   │   └── RepositoryBase, IRepository
│   └── Attributes/
│       └── AccessLogAttribute
├── DevNext/            ← DevNext 固有のコード
│   ├── Controllers/
│   ├── Services/
│   └── Entities/
└── Samples/            ← 各サンプルプロジェクト
    ├── FileSample/
    ├── MailSample/
    └── ...
```

### グレーゾーン判定（3問チェック）

新しいコードを CommonLibrary に入れるか判断するときの基準：

| 質問 | YES → 入れる | NO → 入れない |
|-----|-------------|-------------|
| 他プロジェクトに持っていっても使えるか？ | ✅ | ❌（DevNext 固有 → DevNext に） |
| 単一の責務に収まるか？ | ✅ | ❌（責務が複数 → 分解する） |
| 独立したクラスとして成立するか？ | ✅ | ❌（既存クラスの拡張なら既存を修正） |

## 7. マイグレーション自動実行

### 背景

新規デプロイやローカル環境セットアップのたびに、手動で EF Core マイグレーションを実行するのは：
- 忘れられやすい
- デプロイ手順の複雑さが増す
- チーム開発で初期化がバラバラになる

### 採用理由

```csharp
// Program.cs - アプリ起動時に自動実行
using var scope = app.Services.CreateScope();
var services = scope.ServiceProvider;
var context = services.GetRequiredService<SiteDbContext>();

await context.Database.MigrateAsync();  // マイグレーション適用
await SeedAsync(context);                // 初期データ投入
```

**利点**:
- デプロイ時にマイグレーション実行を忘れない
- ローカル環境セットアップが `dotnet run` だけで完成
- テスト環境の初期化が自動

## 8. ドキュメント自動生成

### 背景

設計書を手作業で更新すると：
- コード変更を忘れて記載する
- スクリーンショットが古いまま
- 実装とドキュメントがズレる
- 保守の負荷が大

### 採用理由

DevNext では `/export-docs` スキルで以下を自動生成：

| ドキュメント | 生成対象 |
|------------|---------|
| 基本設計書 | コントローラー・アクション一覧 |
| 詳細設計書 | エンティティ・サービス設計 |
| DB設計書 | テーブル定義・カラム仕様 |
| 画面設計書 | ビュー一覧・操作フロー |
| API仕様書 | Ajax エンドポイント |

**メリット**:
- コード追加時に自動的にドキュメントが最新に
- 手作業のミスを削減
- ポートフォリオとして「品質の高いドキュメント」を示せる

## 9. テスト戦略

### 背景

テストなしで開発すると、バグが本番環境で見つかります。

### 採用理由

| テスト対象 | 例 |
|-----------|-----|
| **認可（IDOR）** | Admin でない者が他人のリソースにアクセス → 403 返す |
| **バリデーション** | 必須項目なしで送信 → 400 エラー |
| **CSRF 保護** | トークンなし POST → 400 拒否 |
| **ロール別権限** | Member が Admin 専用操作 → 403 |

### テストフレームワーク

```bash
# xUnit + Moq
dotnet test
```

**選定理由**: .NET 標準、EF Core テスト時にメモリDB使用可能、Moq で依存をモック可能

## 10. 開発フロー自動化

### 背景

開発からデプロイまで手作業が多いと：
- チェック漏れが多発
- デプロイのたびにエラーが発生
- 属人化する

### 採用理由

| ツール | 役割 |
|--------|------|
| `/verify` スキル | ビルド＋テスト実行 |
| `/add-page` スキル | ホーム・ナビ・README・ドキュメント更新 |
| `/export-docs` スキル | 設計書を Word/Excel 生成 |
| agent-browser | 実装後の動作確認を自動化 |

**メリット**:
- 「成果物が漏れる」が起こらない
- CI/CD 前の品質チェックが自動
- 新人でも同じレベルの品質で開発できる

## まとめ: 設計の 3 原則

### 1. セキュリティデフォルト
設計段階で認可・CSRF・XSS 対策を組み込む。後付けは漏れる。

### 2. 再利用性重視
新案件で使えるコード（CommonLibrary）と固有コード（DevNext）を明確に分離。

### 3. 保守性優先
「実装のしやすさ」より「理解のしやすさ」。テストできる設計にする。

## 面接での説明例

**質問**: 「このプロジェクトで工夫した設計的な点は？」

**回答例**:

> DevNext は Web テンプレートですが、3 つの設計判断を重視しました。
>
> 1. **セキュリティをフレームワークに任せる**: パスワード管理・CSRF・IDOR 対策を ASP.NET Core Identity に任せることで、実装ミスを減らしています。
>
> 2. **責務を明確に分離**: リポジトリパターンとDIパターンでデータアクセスとビジネスロジックを分ける。テストが書きやすく、新案件への転用も容易です。
>
> 3. **自動化で品質を上げる**: `/verify` `/add-page` などスキルを組み込み、チェック漏れを減らす。ドキュメント自動生成で「実装とドキュメントのズレ」も防止しています。
>
> 特に IDOR 対策は「ログイン確認」と「所有権確認」を分けることで、認可漏れを防止しています。
